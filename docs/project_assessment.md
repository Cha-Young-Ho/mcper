# MCPER 프로젝트 종합 평가 (2026-03-30)

프로젝트 전체 코드 구조 및 각 파트별 평가 기록.

---

## 📋 프로젝트 개요

- **스택**: FastAPI + FastMCP + PostgreSQL(pgvector) + Redis + Celery
- **규모**: ~7,300줄 (93개 Python 파일)
- **핵심 기능**: 기획서(Spec) RAG 검색 + 버전화 규칙 시스템 + MCP 도구 서빙
- **배포**: Docker Compose / Kubernetes 준비됨

---

## 1️⃣ 데이터베이스 계층 ⭐⭐⭐⭐

**평가**: 잘 설계됨

**강점:**
- SQLAlchemy ORM 명확함 (Spec, Rule variants, Auth, RAG models)
- pgvector 통합으로 벡터 검색 인프라 준비됨
- 테이블 간 관계가 깔끔 (cascade delete 설정)
- 13개 테이블로 모든 도메인 커버

**약점:**
- 마이그레이션 자동화 (Alembic vs 수동 SQL 불명확)
- Schema versioning 시스템 부재

**개선 대상:**
- Alembic 마이그레이션 구조화
- DB 변경 이력 추적 메커니즘

---

## 2️⃣ 규칙 시스템 ⭐⭐⭐⭐⭐

**평가**: 핵심 기능, 매우 잘됨

**강점:**
- Append-only 버전 관리 (삭제·수정 없음, 롤백 가능)
- Global → Repo → App 3단계 병합 (우아함)
- 1,122줄 `versioned_rules.py` 에 118개 함수로 완전히 구현
- MCP 도구 + Admin UI 통합으로 접근성 높음

**약점:**
- 없음 (설계 우수)

**테스트 대상:**
- 규칙 버전 자동 증가 로직
- 3단계 병합 시 우선순위
- Rollback 기능 정합성

---

## 3️⃣ RAG 및 검색 ⭐⭐⭐

**평가**: 기본 구조는 좋으나 불완전

**강점:**
- 벡터 + FTS 하이브리드 검색 구현 (search_hybrid.py)
- 임베딩 백엔드 플러그인 (local/OpenAI/Bedrock)
- CodeNode 테이블 준비됨
- 검색 성능 최적화 (pgvector 인덱싱 가능)

**약점:**
- **CodeNode 자동 인덱싱 파서 부재** (AST 크롤러 없음) 🔴
- 임베딩 실패 시 복구 메커니즘 부재
- SpecChunk 벡터 업데이트 전략 모호

**개선 필수:**
1. AST 기반 CodeNode 자동 생성 파서
2. 임베딩 실패 → 폴백 (FTS only) 또는 재시도 전략
3. 임베딩 차원 변경 시 전량 재생성 체크

**테스트 대상:**
- 벡터 검색 실패 시 FTS 폴백
- 임베딩 백엔드 전환 (dim 변경)
- 대용량 스펙 청킹 (메모리)

---

## 4️⃣ 관리자 UI ⭐⭐⭐

**평가**: 기능 풍부하나 코드 복잡

**강점:**
- HTMX 스트리밍으로 반응성 좋음
- 60+ 엔드포인트로 전체 기능 커버
- 규칙·스펙·도구 통합 관리
- HTML 템플릿과 API 로직이 분리됨

**약점:**
- **`admin.py` 1293줄** (단일 파일로 너무 큼) 🔴
- 모듈 분리 필요 (규칙, 스펙, 도구별로)
- 템플릿 재사용성 낮음

**개선 필수:**
1. admin.py 모듈 분리:
   - `routers/admin_rules.py` (규칙 관리)
   - `routers/admin_specs.py` (스펙 관리)
   - `routers/admin_tools.py` (도구 카탈로그)
2. 공통 템플릿 추출 (카드, 모달, 리스트)

**테스트 대상:**
- require_admin_user 인증
- HTMX 스트리밍 응답 형식
- 각 엔드포인트 CRUD 정합성

---

## 5️⃣ 인증/보안 ⭐⭐

**평가**: 기본만 있고 운영 준비 부족

**강점:**
- JWT + OAuth2 구조 (Google, GitHub SSO 준비)
- Host 헤더 화이트리스트
- API 토큰 모델 (ApiKey) 정의

**약점:**
- **Admin 기본 패스워드 "changeme"** (경고만, 강제 없음) 🔴 **CRITICAL**
- API 토큰 만료 검증 확인 필요
- CORS/CSRF 미흡
- Rate limiting 부재

**개선 필수:**
1. Admin 패스워드 초기화 강제 (런타임 체크)
2. API 토큰 만료 검증 (auth/dependencies.py)
3. CSRF 토큰 + SameSite cookie
4. Rate limiting (FastAPI middleware)

**테스트 대상:**
- Admin 토큰 만료 검증
- CORS Origin 검증
- OAuth 콜백 보안 (state 파라미터)
- 기본 패스워드 사용 불가 강제

---

## 6️⃣ 백그라운드 작업 (Celery) ⭐⭐⭐

**평가**: 기본 구조 있으나 모니터링 부족

**강점:**
- 청크 인덱싱 비동기화
- 재시도 3회 설정
- Celery + Redis 연동

**약점:**
- 실패 스펙 추적 기능 없음
- 모니터링 대시보드 부재
- Redis 장애 시 fallback 없음
- 작업 큐 깊이 모니터링 부재

**개선 필수:**
1. 실패 스펙 DB 테이블 (`failed_specs`)
2. Celery Flower 대시보드
3. /health/rag 엔드포인트 확장 (큐 깊이, 작업 상태)
4. Redis 연결 끊김 감지

**테스트 대상:**
- 임베딩 실패 재시도
- Redis 브로커 다운 시 대응
- Celery 작업 타임아웃
- 큐 적체 감시

---

## 7️⃣ 설정 관리 ⭐⭐⭐⭐

**평가**: 유연하고 잘 설계됨

**강점:**
- YAML + 환경변수 계층화 (우선순위 명확)
- `${VAR}` / `${VAR:-default}` 치환 지원
- Feature flag (AUTH, ADMIN, MCP on/off) 우아함
- 다중 임베딩 백엔드 선택 가능
- Pydantic 검증으로 타입 안전성

**약점:**
- 환경 검증 로직이 약함 (optional 설정 오류 발견 어려움)

**개선 사항:**
1. 필수 환경변수 검증 강화
2. Config 변경 시 재검증 메커니즘
3. Secrets 관리 (예: AWS Secrets Manager)

---

## 8️⃣ MCP 도구 ⭐⭐⭐⭐

**평가**: 핵심 기능 완성

**강점:**
- 10+ 도구 (업로드, 검색, 규칙, 코드 분석)
- Transport 보안 (Host/Origin 검증)
- 모든 도구가 MCP 스펙 준수
- 비동기 스트리밍 지원

**약점:**
- 문서화 부족 (mcp_tools_docs.py 는 있으나 부분적)
- 에러 메시지 사용자 친화적 아님

**개선 사항:**
1. 각 도구별 JSON 스키마 명확화
2. 에러 응답 표준화
3. MCP 도구 사용 예시 추가

**테스트 대상:**
- MCP Transport 보안 (Host 검증)
- 각 도구 입력/출력 스키마
- 스트리밍 응답 형식

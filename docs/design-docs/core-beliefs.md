# docs/design-docs/core-beliefs.md — 핵심 설계 원칙\n\n---\n\n## 설계 철학 (Design Philosophy)\n\n### 1. 에이전트 중심 (Agent-Centric)\n\n**원칙**: 모든 기능은 **에이전트 도구 호출**을 중심으로 설계\n\n**의미**:\n- 에이전트가 독립적으로 기획서 검색, 규칙 조회, 코드 분석 가능\n- 수동 개입 최소화\n- 에이전트 학습 곡선 낮음\n\n**구현 예**:\n```python\n# MCP 도구\nasync def search_spec_and_code(query: str, app_target: str) -> dict:\n    \"\"\"기획서 + 코드 하이브리드 검색.\"\"\"\n    # 에이전트가 호출 → 즉시 결과 반환\n```\n\n---\n\n### 2. 단순함 (Simplicity)\n\n**원칙**: 필요한 것만, 정확히\n\n**의미**:\n- 과도한 추상화 회피\n- 명확한 인터페이스\n- 문서로 명시된 제약\n\n**반례** (피해야 할 것):\n- 모든 경우를 대비한 범용 프레임워크\n- 깊은 상속 (다형성)\n- 암묵적 동작\n\n---\n\n### 3. 감시 추적 (Auditability)\n\n**원칙**: 모든 변경을 기록하고 역추적 가능하게\n\n**의미**:\n- Append-only 규칙 (수정/삭제 금지)\n- 타임스탬프 + 버전\n- 감사 로그 (누가, 언제, 뭘 바꿈)\n\n**구현 예**:\n```sql\n-- global_rule_versions\nid | version | body | created_at\n1  | 1       | \"global rule v1\" | 2026-03-01 10:00\n2  | 2       | \"global rule v2\" | 2026-03-02 14:30\n```\n\n---\n\n### 4. 인적 오류 방지 (Fail-Safe)\n\n**원칙**: 사용자 실수를 자동으로 방지\n\n**의미**:\n- 확인 없이 위험한 작업 불가\n- 기본값은 안전한 선택지\n- 되돌리기 (undo) 가능성\n\n**예**:\n```python\n# 규칙 삭제: 불가능 (append-only)\nDELETE FROM global_rule_versions\n# → 에러: \"규칙은 삭제할 수 없습니다\"\n\n# 기획서 삭제: 확인 필요\nDELETE FROM specs WHERE id = ?\n# → UI: \"정말 삭제하시겠습니까? 이 작업은 되돌릴 수 없습니다.\"\n```\n\n---\n\n## 아키텍처 원칙\n\n### 5. 모듈 독립성 (Module Independence)\n\n**원칙**: 각 모듈은 독립적으로 테스트·배포 가능\n\n**구현**:\n```\napp/\n├── db/           # 데이터 계층 (ORM, 스키마)\n├── services/     # 비즈니스 로직 (검색, 임베딩)\n├── routers/      # HTTP 엔드포인트\n├── tools/        # MCP 도구\n└── worker/       # 비동기 작업\n\n각 계층은 하위 계층에만 의존 (역의존 금지)\n```\n\n---\n\n### 6. 암묵적 동작 금지 (No Magic)\n\n**원칙**: 모든 동작이 명시적이고 문서화됨\n\n**의미**:\n- 주석 없이 이해되는 코드\n- 숨겨진 부작용 없음\n- 설정 오버라이드는 명확\n\n**반례**:\n```python\n# ❌ 나쁜 예 (암묵적)\nclass Spec(Base):\n    @property\n    def chunks(self):  # 선택적으로 로드\n        if self._chunks is None:\n            self._chunks = db.query(SpecChunk)\n        return self._chunks\n\n# ✅ 좋은 예 (명시적)\nspec = db.query(Spec).options(selectinload(Spec.chunks)).first()\n```\n\n---\n\n### 7. 명확한 데이터 흐름 (Clear Data Flow)\n\n**원칙**: 데이터의 출처와 변환 과정이 명확\n\n**구현**:\n```\n클라이언트 (JSON) \n  ↓ (Pydantic 검증)\n요청 모델\n  ↓ (비즈니스 로직)\nDB 엔티티\n  ↓ (쿼리 응답)\n응답 모델\n  ↓ (JSON 직렬화)\n클라이언트 (JSON)\n```\n\n각 단계에서 타입이 명확하고, 변환이 명시적\n\n---\n\n## 성능 원칙\n\n### 8. 비동기 우선 (Async-First)\n\n**원칙**: CPU/IO 집약적 작업은 비동기로\n\n**구현**:\n- FastAPI: async/await\n- Celery: 워커 분리\n- 임베딩: 배치 처리\n\n---\n\n### 9. 캐싱 전략 (Caching Strategy)\n\n**원칙**: 읽기 작업은 캐시, 쓰기는 즉시\n\n**구현**:\n```python\n# 규칙 조회: 캐시 (버전 변경 시 무효화)\nrule = cache.get(f\"rule:{app_name}:{version}\")\nif not rule:\n    rule = db.query(...)\n    cache.set(...)\n\n# 규칙 발행: 즉시 DB (캐시 무효화)\ndb.insert(new_rule)\ncache.delete(f\"rule:{app_name}:*\")\n```\n\n---\n\n## 보안 원칙\n\n### 10. 최소 권한 (Principle of Least Privilege)\n\n**원칙**: 각 역할은 최소한의 권한만\n\n**구현** (향후):\n```\n어드민: 규칙 발행, 기획서 삭제\n에이전트: 규칙 조회, 기획서 검색 (읽기만)\n워커: 색인만 (쓰기 제한)\n```\n\n---\n\n### 11. 기본 보안 (Secure by Default)\n\n**원칙**: 보안 설정이 기본값\n\n**구현**:\n- Host/Origin 검증 활성화 (비활성화는 명시)\n- CSRF 토큰 필수\n- HTTPS 권장\n- 환경 변수로 민감정보 분리\n\n---\n\n## 개발 원칙\n\n### 12. 테스트 우선 (Test-Driven)\n\n**원칙**: 기능 구현 전 테스트 작성\n\n**단계**:\n1. 테스트 작성 (실패)\n2. 최소 구현 (통과)\n3. 리팩토링 (개선)\n\n---\n\n### 13. 문서 자동화 (Documentation as Code)\n\n**원칙**: 코드와 문서를 동기화\n\n**구현**:\n```python\n@app.post(\"/api/specs\")\n\"\"\"기획서 생성.\n\nArgs:\n    title: 기획서 제목\n    content: 기획서 본문\n\nReturns:\n    {\"id\": 123, \"created_at\": \"2026-03-30T...\"}\n\"\"\"\nasync def create_spec(req: CreateSpecRequest):\n    # OpenAPI 자동 생성\n    ...\n```\n\n---\n\n## 의사결정 기준\n\n### 기술 선택\n\n**기준** (우선순위):\n1. 안정성 (버그 적음, 커뮤니티 활발)\n2. 성능 (응답 시간, 메모리)\n3. 개발 생산성 (코드 짧음, 배우기 쉬움)\n4. 운영 복잡도 (배포, 모니터링)\n\n**예**:\n- FastAPI: 안정성 (Starlette) + 생산성 (async) ✅\n- SQLAlchemy: 안정성 (오래됨) + 유연성 ✅\n- Celery: 안정성 + 복잡도 (trade-off) ⚠️\n\n---\n\n## 관련 문서\n\n- **docs/DESIGN.md** — 설계 가이드\n- **ARCHITECTURE.md** — 기술 세부사항\n- **docs/dev_log.md** — 작업 로그 (결정 이유)\n"

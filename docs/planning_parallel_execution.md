   ├─ feature/code-parser (CodeNode 파서)

```

### PR 전략

```
Phase 1:
├─ Mon: PR #1 (Admin PW) → 병렬 리뷰
├─ Tue: PR #2 (API 토큰) → 병렬 리뷰
├─ Wed: PR #3 (CORS/CSRF) → 병렬 리뷰
└─ Thu: Merge all → main (또는 staging)

Phase 2:
├─ Fri: PR #4 (admin 분리) → 리뷰
├─ Mon: PR #5 (CodeNode) → 리뷰
├─ Wed: PR #6 (Celery) → 리뷰
└─ Tue: Merge all → main
```

### Merge Conflict 방지

```
전략:
1. 각 PR은 분리된 파일/함수 수정 (동일 파일 최소화)
   ✅ admin.py 분리: 각 PR이 새 파일 추가 (충돌 X)
   ✅ CodeNode: 기존 rag_tools.py 확장만 (충돌 최소)

2. 타이밍: 병렬 PR이 다른 PR에 의존 시 순차 merge
   예: Phase 2 PR들은 Phase 1 merge 후 시작

3. Test CI: 모든 PR에 자동 테스트 실행
   └─ 충돌 발견 → 조기 경고
```

## 🛡️ 위험 완화

### Risk 1: @coder 병목

**위험**: @coder 1명만 구현 → Phase 2 병렬 불가

**완화**:
```
A) Phase 1 단축 (Mon-Thu)
   └─ 항목 1, 2, 3 병렬 → 짧은 시간에 완료

B) Phase 2 지연 허용
   └─ 항목 4, 5, 6 순차 진행 (리스크 낮음)

C) 외부 리소스 (선택)
   └─ 프리랜서 @coder 추가 (항목 4 담당)
```

### Risk 2: 테스트 병목 (@tester 1명)

**위험**: 테스트 작성 지연 → 코드 검증 안 됨

**완화**:
```
A) 테스트 자동화
   └─ GitHub Actions + pytest 자동 실행

B) @tester 우선순위
   └─ 보안 항목 1, 2, 3 먼저 테스트
   └─ 구조 항목 4, 5, 6 나중 (리스크 낮음)

C) @coder 셀프 테스트
   └─ 기본 단위 테스트는 @coder가 작성
   └─ @tester는 E2E/통합 테스트만
```

### Risk 3: 데이터베이스 변경 충돌

**위험**: 동시에 6개 항목이 DB 수정 → 마이그레이션 충돌

**완화**:
```
A) 테스트 DB 격리
   └─ 각 테스트는 트랜잭션 롤백

B) 마이그레이션 순차 적용
   └─ Phase 1 → Phase 2 merge 시
   └─ SQL ALTER 순서 정의

C) Alembic 검토 (향후)
   └─ 현재: 수동 SQL (낮은 리스크)
```

### Risk 4: 코드 리뷰 병목

**위험**: 6개 PR 동시 리뷰 → 검수 지연

**완화**:
```
A) 리뷰 우선순위

## 제품 비전

**Spec MCP**: 에이전트(Cursor/Claude) 중심의 협업 환경

### 핵심 가치

1. **에이전트 자율성**
   - 도구 호출로 정보 검색, 기획서·규칙 수정
   - 에이전트가 독립적으로 의사결정

2. **감시 추적**
   - 모든 변경을 버전으로 기록 (append-only)
   - 누가, 언제, 무엇을 바꿨는지 추적

3. **간결함**
   - 최소한의 의존성
   - 한눈에 이해되는 구조

---

## 사용자 정의

### 1. 개발자 (에이전트)

**목표**: 빠르게 컨텍스트를 찾고 규칙을 따르기

**필요한 것**:
- 기획서 검색 (의미론적 + 키워드)
- 규칙 조회 (프로젝트별)
- 코드 분석 (영향도, 의존성)

**통증**:
- 기획서가 산재 (Notion, Confluence, Slack)
- 규칙이 변경되어도 에이전트가 모름
- 코드 변경 영향도를 추측해야 함

**해결**:
- MCP 도구로 원클릭 검색
- 규칙 버전 자동 확인
- CodeNode 그래프 분석

---

### 2. 시니어 개발자 (PM/리뷰어)

**목표**: 기획서·규칙을 관리하고 팀의 일관성 유지

**필요한 것**:
- 어드민 UI (기획서·규칙 편집)
- 규칙 버전 이력 조회
- 대시보드 (MCP 호출 통계)

**통증**:
- 어디서 규칙을 수정해야 하는지 불명확
- 규칙 변경 이력을 못 추적
- 어떤 에이전트가 뭘 썼는지 몰라

**해결**:
- 중앙집중식 어드민
- Append-only 규칙 시스템
- MCP 도구 호출 로깅

---

### 3. 운영자 (DevOps/인프라)

**목표**: 안정적이고 확장 가능한 인프라 운영

**필요한 것**:
- 헬스체크 + 모니터링
- 자동 확장
- 재해 복구

**통증**:
- 임베딩 모델 느려서 큐가 밀림
- 대량 업로드 시 DB 연결 고갈
- 에러가 나도 원인을 모름

**해결**:
- Celery 큐 모니터링
- 커넥션 풀 자동 조정
- 구조화 로깅 (에러 추적)

## 경쟁 우위

### vs. 규칙 관리 시스템 (CLIP, Rule Engine)

| 항목 | Spec MCP | 다른 도구 |
|------|---------|---------|
| 에이전트 통합 | ✅ MCP 네이티브 | ❌ API 또는 파일 |
| 버전 관리 | ✅ Append-only | ⚠️ 충돌 해결 어려움 |
| 실시간 반영 | ✅ (버전 체크) | ⚠️ 캐시 무효화 필요 |
| 코드 연결 | ✅ CodeNode | ❌ 미지원 |

### vs. 문서 협업 (Notion, Confluence)

| 항목 | Spec MCP | Notion |
|------|---------|--------|
| 에이전트 검색 | ✅ API + MCP | ❌ 수동 검색 |
| 오프라인 | ✅ 로컬 가능 | ❌ 클라우드만 |
| 프라이버시 | ✅ 자체 호스팅 | ❌ 외부 의존 |
| 협업 UI | ⚠️ 어드민만 | ✅ 풍부함 |

## 전략적 로드맵

### Phase 1: 기초 강화 (2026 Q2, 현재)

**목표**: 프로덕션 준비

**핵심 작업**:
1. 보안 강화 (CRITICAL 3개)
2. admin.py 모듈화
3. CodeNode 파서
4. Celery 모니터링

**KPI**:
- 보안 취약점 0 (CRITICAL)
- 테스트 커버리지 >= 30%
- 배포 가이드 완성

**예상 기간**: 7주

---

### Phase 2: 확장성 (2026 Q3)

**목표**: 엔터프라이즈급 안정성

**핵심 작업**:
1. 테스트 커버리지 50% 이상
2. 모니터링 대시보드 (Grafana)
3. 자동 확장 (K8s)
4. 레이트 제한 + WAF

**KPI**:
- 99.9% 가용성 (SLA)
- p95 응답 시간 < 500ms
- 동시 사용자 100+

**예상 기간**: 4주

---

### Phase 3: 기능 확장 (2026 Q4)

**목표**: 크로스팀 협업 플랫폼

**핵심 작업**:
1. 권한 관리 (RBAC)
2. 조직별 규칙 격리
3. 감사 로그 (Compliance)
4. Webhook (외부 시스템 연동)

**KPI**:
- 3개 조직 온보딩
- 500명 이상 사용자
- SOC 2 Type II 인증

**예상 기간**: 6주

## 의사결정 기준

### 새 기능 추가 시

```
┌─ Core path인가? (기획서, 규칙, 검색)
├─ Yes → 우선순위 논의
└─ No  → "나중에" 또는 기각

┌─ 사용자 2명 이상 요청?
├─ Yes → 기획 검토
└─ No  → 백로그

┌─ 유지보수 부담이 적은가?
├─ Yes → 진행
└─ No  → 아키텍처 검토 필요
```

### 기술 선택

```
┌─ 기존 스택에 있는가?
├─ Yes → 사용 (학습 곡선 낮음)
└─ No  → 아래 기준 검토

┌─ 커뮤니티가 크고 활발한가?
├─ Yes → 고려
└─ No  → 회피

┌─ 우리가 유지보수할 수 있는가?
├─ Yes → 결정
└─ No  → 외주 또는 회피
```

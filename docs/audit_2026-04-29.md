## 5. 권장 진행 순서

### Phase 2 (다음, 자동화 툴만) — 위험 낮음

1. `ruff check --fix` — 미사용 import, 스타일 자동 수정
2. `ruff format` — 포맷팅 통일
3. `bandit -r app/` — 보안 스캔 결과 리포트
4. `pip-audit` — 취약 의존성 리포트
5. `mypy app/` — 타입 오류 리포트 (이번엔 자동 수정 X, 리포트만)

**예상 산출물**: 1~2 커밋. 의미 변경 없는 개선만.

### Phase 3 (스케일/LB 대비) — 위험 중간

**우선순위**:
1. L01, L02 Redis 외부화 (가장 급함, P0)
2. L06 PgBouncer + 풀 튜닝
3. L04 임베딩 외부화 (메모리 문제)
4. L09 헬스체크 분리
5. Caddy sticky session 설정 문서화

각 항목 독립 커밋. 컨테이너 재기동 수반.

### Phase 4 (품질 리팩터링) — 영역별

**감사 결과 기준 우선순위**:
1. Q01, Q02 공통 헬퍼 추출 (30분, 낮은 리스크)
2. Q04 라우터 → 서비스 분리 (크기 큼, 라우터 하나씩)
3. Q03 거대 파일 분할 (리스크 있음, 단계적)
4. S01~S06 보안 P0/P1 수정 (Phase 3 와 병행 가능)
5. P01, P02 N+1 제거 (성능 즉시 개선)

## 6. 이번 감사에서 **하지 않기로 한** 것

- **모든 P2 자동 수정** — 사용자 판단 필요
- **거대 파일 자동 분할** — 라우터 단위 승인 필요
- **DB 스키마 변경** — 프로덕션 영향

위 항목은 사용자가 우선순위 지정 후 개별 세션에서 처리.

## 7. 요약 한 문장

**mcper 는 단일 컨테이너 기준으로 건전하지만, 다중 인스턴스·LB 배치 전에 OAuth/MCP 세션의 Redis 외부화(L01/L02/L03), 임베딩 모델 외부화(L04), 보안 기본값 검증(S01/S02) 을 반드시 선행해야 한다.** N+1 쿼리(P01/P02) 와 공통 헬퍼 추출(Q01/Q02) 은 낮은 리스크로 즉시 개선 가능.

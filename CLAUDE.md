## 공통 규칙 (필수)

1. **컨텍스트 절약** — 파일 읽기 전 추측 금지, 변경 전 확인, 요청 없는 개선 금지, compact 응답
2. **문서화** — 큰 작업 끝나면 `docs/dev_log.md` 에 append (날짜/에이전트/요약/파일/결정사항)
3. **선호 추적** — 파악된 스타일/선호는 `.agents/*/` 의 해당 파일 갱신
4. **협업** — 모호한 요청은 확인 후 진행, 선행 작업은 `dev_log.md` 최신 항목 먼저 읽기

---

## 협업 흐름

```
@pm (타당성) → @planner (기획) → @senior (설계) → @coder (구현)
                                                      ↕ (병렬)
                                                   @tester (테스트)
                                                      ↓
                                                  @infra (검수)
     ↓ (필요시)
  @archivist (대용량 파일 읽기 → 메모 제공)
```

**핵심: @coder 와 @tester 는 설계 완료 후 병렬로 코드 + 테스트를 함께 작성 (상호 협력)**

단순 작업은 중간 단계 생략 가능. 예: 버그픽스는 `@coder` + `@tester` 만.

## No Manual Code 철학 (Harness Engineering)

**규칙**: 프로덕션 코드는 에이전트만 작성. 사용자는 설정·문서·요청만 담당.

- ✅ 프로덕션 코드: 항상 @coder → 작성
- ✅ 테스트 코드: 항상 @tester → 작성
- ✅ 설정/문서: 사용자 직접 작성 가능
- ❌ 사용자가 코드 작성 후 "여기 수정해줘" → 규칙 위반

## 기록관(@archivist) 활용 규칙

**다른 에이전트가 다음 중 하나라도 해당하면 @archivist 에게 먼저 요청:**

- 파일 크기 **1000줄 이상** (admin.py, versioned_rules.py 등)
- 동시에 **5개 이상 파일** 분석 필요
- 최근 `.claude/archivist_notes/` 에 **관련 메모가 있으면 재활용**

**예외: 사용자가 "빨리 해줘", "급해" 등 시간 압박 명시 → 기록관 스킵하고 직접 진행**

기록관이 읽은 메모는 `.claude/archivist_notes/` 에 저장되며, 같은 정보가 필요하면 메모 활용으로 컨텍스트 절감.

## 작업 완료 후 보고서 작성

**모든 에이전트가 큰 작업 완료 후 작성해야 함:**

```markdown
## [날짜]: @[에이전트명] [작업 제목]

### 작업 내용
- 주요 변경 3-5줄 요약
- 파일: [file1.py], [file2.py]

### 판단 이유
- Why: 왜 이렇게 했는가?
- Risk: 예상되는 위험요소

### 결과
✅ 완료 / ⚠️ 부분 완료 / ❌ 보류

### 다음 단계
- 후속 에이전트 또는 의존성
```

**저장 위치:** `docs/dev_log.md` 맨 위에 추가 (최신순)

**작성 기준:**
- 파일 3개 이상 수정
- 로직 변경 (버그픽스 제외)
- 새 기능 추가
- 아키텍처 결정

**상세 템플릿:** `.agents/report_template.md` 참고

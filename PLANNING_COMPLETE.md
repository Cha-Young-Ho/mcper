## 🎯 주요 설계 결정사항

### 1. Admin 패스워드: 메모리 + 환경 변수 조합

```
✅ 선택: 메모리 플래그 (간단)
❌ 대안: DB 테이블 (복잡)

이유:
- 초기화는 1회성 작업
- 프로세스 재시작 시 초기값으로 복구해야 함
- 환경 변수 업데이트 권장 (운영 문서화)
```

### 2. CSRF 토큰: Redis 저장 (프로덕션) + 메모리 (개발)

```
✅ 선택: Redis (확장성, 분산 환경)
❓ 대안: 메모리 dict (개발용, 단일 프로세스)

이유:
- Kubernetes 배포 지원
- 세션 저장소 일원화
- TTL 관리 자동화
```

### 3. CodeNode 파서: Python만 먼저 (Phase 2)

```
✅ 선택: Python AST (ast 모듈, 표준)
❌ 다음: JavaScript tree-sitter (복잡도 높음)

이유:
- Python: 표준 라이브러리, 간단
- JavaScript: 별도 바이너리 필요
- MVP 우선 (1주 예상 vs 3주 예상)
```

### 4. Celery 실패 추적: DB 테이블 (TaskFailure)

```
✅ 선택: 독립 테이블 (조회 성능)
❌ 대안: Redis sorted set (휘발성)

이유:
- Redis 장애 시에도 추적 필요
- 장기 보관 (90일 이상)
- 대시보드 쿼리 성능
```

### 5. admin.py 분리: 4개 라우터 + utils

```
✅ 선택: 기능별 분리 (규칙/스펙/도구)
❓ 대안: 도메인별 분리 (admin/tools/rules)

이유:
- 현재 구조와 일치
- URL 경로 유지 간편
- 엔드포인트 관점이 자연스러움
```
